## What it does

You point FileID at a folder. It reads every file inside it — images, video, PDFs, audio, documents — and builds a single searchable library that understands what's *in* them. People are clustered into named cards. Duplicates are grouped by perceptual hash. A local vision-language model writes captions and proposes filenames. A folder restructure is previewed before anything moves. Nothing leaves the Mac.

## Quickstart

**Requirements**
- macOS 15+
- Apple Silicon (M1 / M2 / M3 / M4)
- Xcode 16+ at `/Applications/Xcode.app`
- ~4 GB free disk (more if you download a VLM)

**Build & launch**

```bash
git clone https://github.com/AdamNolle/FileID.git
cd FileID
./run.sh
```

`run.sh` builds both binaries via `swift build -c release`, assembles `FileID.app` at the repo root, wipes any stale DB to start fresh, and `open`s the app.

**Dev build** (no fresh-state wipe)

```bash
DEVELOPER_DIR=/Applications/Xcode.app/Contents/Developer swift build
```

## The six steps

The user-facing pipeline, in order:

| # | Step | What happens |
|---|------|--------------|
| **1** | **Scan** | Apple Vision finds faces (with capture-quality scores), EXIF, OCR text, perceptual hashes. ~150 files/s on Apple Silicon. |
| **2** | **Embed** | Each face is L2-normalized via **ArcFace iResNet50** (Buffalo-L) on the ANE — the same identity-trained model Immich uses. Quality filter drops blur / profile / tiny faces. |
| **3** | **Cluster** | **Chinese Whispers** on a cosine-kNN graph. No fixed global threshold; local density decides cluster boundaries. Same person at age 5 vs 12 clusters together; siblings stay apart. |
| **4** | **Name & tag** | Name the most-photographed people. One click tags every photo of that person with macOS Finder tags (visible in Finder, Spotlight, Smart Folders). |
| **5** | **Deep Analyze** | Local VLM writes a caption + suggested filename for every photo, using the names you provided. Bulk-rename in one batch with one-click undo. |
| **6** | **Restructure** | Assistant-mode reorganizer keeps your meaningfully-named folders intact, dissolves only the junk-named ones. Symlink mode is reversible; convert to real moves when you're satisfied. |

The splash screen shows this verbatim before you pick a folder.

## Performance

Real numbers from a 60,000-file TrueNAS library on M1 Pro 16 GB:

| Phase | Throughput | Notes |
|-------|------------|-------|
| Discovery (60K) | ~3,400 files/s | NAS-bound; sorts by path for I/O locality. |
| Tagging — image phase | ~150 files/s | 14 parallel workers, ANE-bound. Vision (face + classify) ~80 ms/file, MobileCLIP ~7 ms/file. |
| Tagging — mixed phase | ~75 files/s | PDFs trigger up to 3 OCR passes. Audio / docs / video are near-instant. |
| ArcFace embedding (lazy, post-scan) | ~25 ms / face on ANE (iResNet50), ~10 ms (MobileFace) | Runs in Stage D, bounded to 4 in-flight. 50K faces ≈ 21 min iResNet50 / 8 min MobileFace. |
| Chinese Whispers clustering (50K faces) | ~5 s | kNN graph build via HNSW + ~10 CW iterations to convergence. |
| Deep Analyze (Qwen 2.5-VL 3B) | ~1.5 s / image | Serial, GPU-bound. ~25 hours for a 60K-image whole-library pass. |
| Manual "Verify with AI" (Qwen) | ~1.5 s / pair | On-demand from the Suggested Merges sheet. Disambiguates twins / age gaps. |

Engine throughput is **10× v1's baseline** (which capped at 13.8 files/s). Deadlock-free across hours of continuous scanning.

## Privacy

- **All inference is local.** Apple Vision runs on the ANE. MobileCLIP runs on the ANE. ArcFace runs on the ANE. VLM models run on the GPU via MLX. No image, embedding, caption, or filename ever leaves your Mac.
- **No telemetry, no analytics, no crash reporting.** The structured JSONL log at `~/Library/Application Support/FileID/logs/scan.jsonl` is local-only and `jq`-queryable.
- **Models download from HuggingFace on first use.** That's an HTTPS request to `huggingface.co` for the weights — same as `pip install`. After that, weights live in `~/Documents/huggingface/models/` and are reused offline.

## Usage

### First run

1. **Pick a folder.** Drag-drop onto the window, click **Pick a folder** on the splash, or use the sidebar button.
2. **Click Start Scan.** The sidebar's pipeline indicator (Discover → Tag → Post-scan → Done) goes live; the Library grid populates as files are processed.
3. **Browse.** FTS5 search (filenames + OCR), kind facets (All / Images / Video / Docs / PDFs / Audio), click any tile for a preview sheet with metadata, EXIF, tags, and Reveal in Finder.

### Cleanup

Duplicate groups by perceptual hash, sorted by group size. Per-tile checkboxes, group-level **Select…** menu (all / none / all-except-keeper / invert), per-group **Skip group**, top-bar **Delete N selected** trashes everything checked across the whole view via `FileManager.trashItem(at:)`.

### People

Face cards backed by ArcFace 512-d identity embeddings. **Run face clustering** triggers the full pipeline:

1. **Quality filter** drops faces below 0.40 capture quality, under 0.5 % of frame area, or beyond 50° yaw / 30° pitch — they keep their row but don't anchor a cluster.
2. **ArcFace iResNet50** embeds each remaining face crop (from `face_crops/`) on the ANE.
3. **Chinese Whispers** runs on a kNN cosine graph (k = 20, edge threshold 0.40). Local density decides cluster boundaries — no global L2 threshold to tune.
4. **Tight-pair centroid auto-merge** polishes any residual fragmentation.

Tap a card to see every photo for that person, name them with structured fields (Title / First / Middle / Last / Suffix), or mark **I don't know** to exclude from clustering and Deep Analyze captions. **Verify with AI** in the Suggested Merges sheet runs Qwen on a specific subset when you want a VLM's judgment on borderline cases (twins, age gaps).

Each named cluster has a **"Tag all N photos"** button that writes the person's display name as a Finder tag onto every photo of them. If you later edit the name, an inline **"Replace 'old' with 'new'"** affordance appears so the on-disk tags stay in sync.

No migration ceremony — face recognition uses whichever ArcFace `.mlpackage` is on disk. Run a fresh scan (or click **Run face clustering**) and any face_prints row missing an embedding gets caught up in Stage D's lazy-extraction step.

### Tags (macOS Finder integration)

Every file's preview sheet has a **Finder tags** card — add tags inline; they write straight to the file via `URLResourceKey.tagNamesKey` and show up everywhere macOS shows tags:

- Finder sidebar (filter by tag)
- Spotlight (`tag:Mom` queries)
- Smart Folders (build saved searches around tags)

Bulk-apply paths:
- **Per-person**: People tab → person card → Tag all N photos.
- **Per-photo**: Library tile → preview sheet → Finder tags card.
- **Cleanup auto-tag**: when you trash a duplicate group, the keepers can be auto-tagged `duplicate-resolved` (Settings toggle, default on).
- (Future: multi-select tile mode with bulk tag apply.)

Library tiles show a small tag-count badge top-right when a file has any Finder tags. Tags are reversible — there's no FileID-only state; if you uninstall, your tags stay in Finder.

### Names

Deep Analyze produces a caption + suggested filename per image. To apply them in bulk:

1. Library header shows **Rename suggestions (N)** when N > 0.
2. Click → preview list of `IMG_8473.HEIC → mom_playing_piano.heic`, default-selects all.
3. Above 50 files, you get a confirmation dialog before applying.
4. Apply runs all renames in one pass with collision-safe `_2`, `_3`, … suffixes.
5. The batch is recorded — **Undo last rename** appears in the header until cleared.

The undo refuses to clobber: if you've moved a renamed file again since, that one is skipped (the rest still revert).

### Deep Analyze

Pick a model in **Settings → AI Models — accuracy tier**. The top three are recommended for your Mac's RAM tier:

| RAM | Recommended (in order) |
|-----|------------------------|
| ≥ 32 GB | Gemma 3 12B → Qwen 3-VL 4B → Qwen 2.5-VL 3B |
| ≥ 16 GB | Qwen 3-VL 4B → Qwen 2.5-VL 3B → Gemma 3 4B |
| ≥ 8 GB  | Qwen 2.5-VL 3B → SmolVLM → PaliGemma 3B |

Run on a single file from the preview sheet, or **Analyze entire library** from the Deep Analyze tab. Sleep prevention keeps the engine running with the lid closed (on AC). Caption + suggested name land in the metadata panel — click **Apply** to rename on disk.

### Restructure (assistant mode)

Restructure respects your existing organization. It classifies every source folder into one of three tiers:

| Tier | Examples | What happens |
|------|----------|--------------|
| **Anchor** | `Albert Einstein/`, `2019/`, `Hawaii/` (when most contents share a signal) | Folder + contents stay exactly where they are. |
| **Mixed** | `Marie Curie's Laboratory/` with 2 Curie photos + 1 stray Einstein | Folder kept; only the outlier moves to its proper bucket. |
| **Junk** | `Untitled folder`, `DCIM`, `Camera Roll`, `Photos (1) (copy)` | Dissolved. Contents re-bucket via the heuristic below. |

For files that DO move (Mixed outliers and Junk contents), the heuristic destination is:

```
People/Mom/2018/             ← named face clusters
Places/37.5_-122.0/2019/     ← GPS-bucketed
Documents/2023/              ← PDFs / OCR-heavy
Photos/2020/05-May/          ← year/month fallback
Misc/                        ← uncategorizable
```

VLM-suggested filenames become the new basenames during the move. The diff preview groups by impact (Stays Put count, Reorganized count, Moved count) so you only see what's actually changing. **Symlink mode** (default) leaves originals in place — try the new layout risk-free, then **Convert to real moves** when you're satisfied.

The folder classifier is a deterministic, unit-tested ~150-LOC pure-Swift module (`FolderClassifier.swift`) — it's not LLM-driven. Stable across re-runs given the same library state.

# Architecture & pipeline deep-dive

The rest of this document is for contributors and the curious. If you just want to use FileID, you can stop reading here.

## Process model

FileID is a **split-process** application. A SwiftUI viewer (`FileID.app`) spawns a Swift CLI engine (`FileIDEngine`) as a child process and supervises it. The two communicate over **stdin/stdout newline-delimited JSON**.

```
┌──────────────────────────────────────────────────────────────────┐
│                  FileID.app   (SwiftUI viewer)                   │
│                                                                  │
│  Library · Deep Analyze · Cleanup · Restructure · People · …     │
│  ReadStore: read-only GRDB queue over the same SQLite WAL        │
│  EngineClient: spawns + supervises the child, reads events,      │
│                exposes @Observable state                         │
└─────────────────────────────────┬────────────────────────────────┘

┌─────────────────────────────────┴────────────────────────────────┐
```

**Why split-process.** The engine owns long-running ML work and database mutation. Killing the UI never blocks the scan; an engine crash auto-respawns (3 attempts in a 60s window with 1s / 4s / 16s backoff) without taking the UI with it. The app is a pure read + UI layer with no SwiftData, no inference, and no schema knowledge beyond what `FileIDShared` exposes.

## Stage A — Discovery

`FileManager.enumerator` walks the root, batched 1024 files per `next()` call. After enumeration completes, the file list is sorted by path lexicographically — this gives I/O locality on NAS, cutting seek thrash on 60K-file libraries by several seconds. Filter by extension and size cap, then push each `DiscoveredFile` into a bounded `AsyncChannel`.

## Stage B — Tagging

`withTaskGroup` spawns N workers where N = `Hardware.workerCap` (formula: `P + E + max(1, P/2)` — 14 on M1 Pro). Each worker pulls from `discoveredChannel.next()` until the channel closes.

Per-file work has explicit pool boundaries:

| Stage | Pool | Why |
|-------|------|-----|
| Decode + dHash + EXIF + doc heuristics | CPU (cooperative pool) | Limited by core count organically; no semaphore needed. |
| Apple Vision (face rects, OCR, classify, saliency) | ANE, `AsyncSemaphore(value: 3)` | ANE is shared hardware. 14 concurrent requests thrash; 3 in-flight saturates without contention. |
| MobileCLIP S2 image embed | ANE, `AsyncSemaphore(value: 2)` | Separate budget so Vision and CLIP don't starve each other. |
| Per-face `VNGenerateImageFeaturePrintRequest` | ANE, same `AsyncSemaphore` as Vision | Capped 5 faces/file, sorted by area, min 0.5 % area. |

**Pre-warm.** Before the worker pool starts, the engine eagerly loads MobileCLIP, instantiates one VisionWorker (which compiles `VNRequest`s), and runs a 1×1 dummy image through both. This pays the first-call ANE warm-up cost once at scan start, not 14 times concurrently mid-scan.

**The Vision deadlock workaround.** Apple's `VNImageRequestHandler.performRequests` deadlocks indefinitely on certain inputs — it's a synchronous GCD wait inside Apple's `VNControlledCapacityTasksQueue` that no Swift Task cancellation can reach. v1 stalled at ~13K files. v2 wraps every Vision call in a `DispatchSemaphore`-based watchdog that abandons the call after 10 s and frees the worker. The orphan thread keeps running but never blocks the pipeline. **This single fix is what made v2 deadlock-free across hours of continuous scanning.**

## Stage C — DB writer

A single dedicated task owns the `DatabasePool`. Calls go through this task; nothing else writes. Workers push `TaggedFile`s into a bounded channel; the writer pulls from `taggedChannel.next()` and buffers up to **100 files OR 50 ms** (whichever fires first), then opens one transaction and inserts the batch.

A single transaction commits: the `files` row, all `tags` rows, all `face_prints` rows, and the `clip_embeddings` blob. A `BatchSummary` IPC event is emitted after each commit.

This is the architectural fix for v1's actor-funnel bottleneck. The DB writer is still serial, but it's serial *over batches of 100* — not per-file. SQLite WAL handles 1000+ tx/s on M1, so the bottleneck moves back to where it should be: the ANE.

## Stage D — Post-scan

Runs sequentially after the tagging channel drains:

1. **Orphan sweep.** `SELECT id, path_text FROM files WHERE scanned_at < session_start`, stat each path off the writer thread, batch-delete rows whose file no longer exists. Capped at 5000 paths/scan; `ON DELETE CASCADE` handles `tags` / `ocr_text` / `face_prints` / `clip_embeddings`.

2. **Lazy face embedding.** For every `face_prints` row that has a bbox but no `arcface_embedding` yet, the engine opens the source image (or reuses the cached `face_crops/<id>.jpg`), runs **ArcFace** on the cropped face on the ANE, and writes the L2-normalized 512-d float32 result back. Bounded to 4 concurrent inferences. Quality-filtered faces are skipped at this stage — they keep their bbox + crop but don't get embedded.

3. **Chinese Whispers clustering.** Pure-Swift implementation (`ChineseWhispers.swift`, ~150 LOC). For each face we query its 20 nearest cosine neighbors via `HNSWIndex`, build a symmetric kNN graph with edges ≥ 0.40 cosine similarity, and iterate up to 20 rounds of "reassign each node to the cluster with the highest summed incoming edge weight." Linear-time in edges; converges in ~10 iterations on real libraries. **No fixed global threshold** — local density decides where one person ends and another begins, which is why same-person-different-conditions clusters together while siblings stay apart.

4. **Tight-pair centroid auto-merge** (polish). The plan-doc's safety net: pairwise centroid L2 over the post-CW persons; merge any two whose centroids are within 0.40, or within 0.50 if at least one is single-photo. Catches residual fragmentation. Cheap on the small post-CW count.

5. **Deep Analyze** (queued, opt-in). Runs sequentially via the `JobQueue` actor — never overlaps with scan or clustering. Single-file Deep Analyze from the preview sheet enqueues alongside.

6. **Manual VLM verification** (opt-in). The Suggested Merges sheet has a **Verify with AI** button that runs Qwen on the user's selected pairs only. Not auto-fired. With ArcFace + Chinese Whispers as the primary clustering pipeline, the borderline band is small and dominated by genuinely-ambiguous cases (twins, age gaps, family resemblance) where a VLM is the right tool.

## Database schema

```sql
files          -- one row per file. wide-but-flat; no JOINs on the hot path
  id, bookmark, path_text, path_hash, size_bytes,
  created_at, modified_at, scanned_at,
  kind, extension, phash, aesthetic,
  has_faces, has_text, camera_model, location_lat, location_lon,
  failed, error_message,
  vlm_description, vlm_proposed_name, vlm_model, vlm_analyzed_at

tags           -- file_id, tag, source (vision|clip|exif|user), score
face_prints    -- id, file_id, person_id, bbox,
                  print_data (legacy Vision feature print, dormant after migration),
                  arcface_embedding (BLOB, 512-d float32, L2-normalized),
                  face_quality (0..1), excluded (1 if filtered out of clustering)
ocr_text       -- file_id (PK), text
ocr_fts        -- FTS5 external-content index over ocr_text + filenames
persons        -- id, name (legacy),
                  title, first_name, middle_name, last_name, suffix,
                  is_unknown, representative_face_id, file_count, created_at
clip_embeddings-- file_id, embedding (BLOB), model
scan_sessions  -- id, root_path, started_at, completed_at,
                  last_file_index, total_files, status

PRAGMA journal_mode = WAL
PRAGMA synchronous = NORMAL
PRAGMA wal_autocheckpoint = 10000
PRAGMA mmap_size = 268435456
PRAGMA temp_store = MEMORY
```

The engine owns a `DatabasePool` (writer); the app opens its own `DatabaseQueue` (read-only, briefly writable for Cleanup row deletes + Restructure path updates). SQLite WAL allows this: concurrent readers never block the writer, and a separate-process brief writer doesn't conflict because no scan is typically active during a Cleanup or Restructure action.

## IPC protocol

The protocol surface lives in `shared/Sources/FileIDShared/IPCProtocol.swift` — Codable types, encoded one JSON object per line.

**Commands** (app → engine):

```
startScan, pauseScan, resumeScan, cancelScan, shutdown,
runFaceClustering, runVLMFaceVerification(modelKind:),
deepAnalyzeFile(...), deepAnalyzeAll(...), deepAnalyzeCancel
```

**Events** (engine → app):

```
ready(EngineInfo)         — engine is up; sends pid, workerCap, RAM
progress(ScanProgress)    — 1 Hz tick: phase, total, processed, files/s, ETA
phaseChanged              — phase transitions (discovering → tagging → …)
discoveryComplete         — total file count locked
fileDone                  — per-file completion (debug detail)
batchSummary(BatchSummary)— per-DB-batch profiler block
scanComplete              — clean terminal event
faceClusteringComplete(...)— person count, faces matched, duration
vlmFaceVerificationProgress / Complete
deepAnalyzeProgress / FileDone / Complete
modelDownloadProgress     — surfaced in Settings + Deep Analyze
queueState(QueueState)    — current running + pending job list
error(EngineError)        — surfaced as a sidebar pill
log                       — debug breadcrumbs
```

**Why stdin/stdout JSON, not XPC.** Zero ceremony, trivially debuggable (`./fileidd | jq .`), no `SMAppService` Login Items dance, works identically in dev and release. XPC remains a future option behind the same `Codable` surface if we add background folder watching that needs to outlive the app.

## Concurrency model

- **App side**: `@MainActor`-bound `EngineClient` is `@Observable`; SwiftUI views observe `state`, `lastProgress`, `lastBatch`, `queueState`, etc. The GCD `readabilityHandler` on the engine's stderr pipe bounces decoded events back to MainActor via `Task { @MainActor [weak self] in ... }`. A `MutexBox<Data>` protects the line buffer for partial reads.
- **Engine side**: `swift-async-algorithms` `AsyncChannel` for backpressure between stages. `withTaskGroup` for worker concurrency. `actor` for the `Database`, `JobQueue`, `JSONLog`. `AsyncSemaphore` (custom — actor-isolated counter, no `defer { Task { ... } }` race) for ANE budgets.
- **MainActor mirrors**: pause and cancel flags use `nonisolated(unsafe) static var` + load/store, readable from inside the synchronous Vision callbacks where MainActor isolation isn't reachable.

## Tech stack — locked decisions

| Layer | Choice | Rationale |

## Engine supervision

- **Auto-respawn.** When the engine exits (kill -9, OOM, panic), the app schedules up to 3 respawn attempts with 1 s / 4 s / 16 s backoff over a 60 s window. A successful `.ready` event resets the budget. After exhaustion, state goes `.crashed` and the user can hit **Restart Engine** in Settings.
- **Clean-exit suppression.** When the app calls `shutdown()` *or* the last engine event was a "work complete" signal (`scanComplete` / `deepAnalyzeComplete` / `faceClusteringComplete`), the next exit is treated as expected — no error pill, no auto-respawn. This covers MLX's known SIGSEGV-in-static-destructor on clean exit (a libmlx bug; harmless, the data is already on disk).
- **Parent-death watchdog.** The engine polls `getppid()` every 5 s; when it returns 1 (parent died, reparented to launchd), the engine flushes its log and `_exit(0)`s. Belt-and-suspenders to the stdin-EOF path, which can miss force-quit on macOS.

## Crash recovery

Every successful batch commit writes the high-water mark (`last_file_index`) to `scan_sessions`. On engine startup, sessions with `status = 'running'` are detected and marked `crashed` with a recovery log line; the cursor is read so a future resume reads from the right position. (The actual resume-from-cursor flow is on the deferred list — the cursor is correct, the app just doesn't yet skip already-processed files.)

The `pHashIndex` is reconstructible from the DB on demand — no separate disk format.
