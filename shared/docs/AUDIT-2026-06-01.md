# FileID Windows — Full Audit (2026-06-01)

**Executive summary.** Four adversarially-verified audit streams (Rust engine static, C# WinUI static, macOS parity, and a live RTX 2060 on-hardware run) produced **618 raw findings** (engine 183, app 201, parity 234), of which **153 were adversarially confirmed** in the original audit passes (engine 48, app 39, parity 66) alongside the on-hardware telemetry. The synthesis additionally retained the P1/P2 "uncertain — manual triage" items, giving a working set of roughly **~50 P1-class (5 cross-platform/integrity blockers after HW-1 was reclassified as unverified), ~150 P2, ~67 P3**. De-duplication folds the same root issues that recur across streams (v13 migration, the SFace/ArcFace 128-d-vs-512-d face-embedding mismatch, the `vlm_model`/`source=` string drifts, CLIP under-batching, the `find_merge_suggestions` O(n²) scans). The five most important issues, in order: (1) **cross-platform DB does not round-trip** — Windows registers `v13_face_verification_anchors` but macOS stops at v12, so a Windows-written library fails to open on macOS (PAR-1/PAR-101 + ENG-3); (2) **the SFace face-embedding pipeline is silently incompatible** between platforms (Windows writes aligned 128-d SFace into `arcface_embedding`, macOS reads it as 512-d ArcFace and never aligns — PAR-2/PAR-31/PAR-32 + ENG-69); (3) **on-hardware throughput is only 4.9 files/s even on CUDA vs the ≥140 target (~29×)** — the CLIP stage barely batches (avg ~1.5 images/dispatch) and per-file wall time (~1.5 s) is ~4× the summed active stage work (~0.36 s), a serialization stall — with peak RSS 5.7 GB vs the 1.5 GB cap (HW-2/HW-3); (4) **an NTFS `file_ref` with the high bit set aborts the entire scan** via a `u64 > i64::MAX` rusqlite panic, losing the whole catalog batch (ENG-18); (5) **silent data-loss on rescan/wipe** — `wipe_all` leaks `foreign_keys=OFF` on the single writer connection on any error path (ENG-2), and rescans that lose faces/OCR/clip leave stale orphan rows (ENG-21). Tag chips, captions-analyzed state, and timestamps also fail to round-trip cross-platform (PAR-78/PAR-85/PAR-88), and the Windows app lacks the macOS RAM-fit gating that prevents OOM-killing the engine with an oversized VLM (PAR-57).

## Methodology

The audit ran **four independent streams**, each split into review units and adversarially verified (a second pass re-checked every claim against the live code; findings whose claimed severity or mechanism did not hold were down-graded to `uncertain` and flagged "manual triage required" rather than dropped). Roughly **~675 review agents** participated across the streams.

**Throughput root cause — PROFILED (2026-06-01), not guessed.** Two hypotheses were tested on hardware and **disproven**: widening the CLIP fill window 20→75 ms barely moved the batch (1.5→2.0) and didn't lift throughput (reverted); and DBWriter back-pressure isn't it (the `out_tx` channel buffers 256). Per-stage instrumentation (`ramplus_us` + `vision_wait_us`, now permanent in `[STATS]`) then pinned it exactly: **RAM++ Swin-L @384 costs ~670 ms/file and runs on only `pool_size=2` sessions** (VRAM-clamped on the 6 GB card), so each worker spends **~680 ms just waiting** for the vision-sem/RAM++ pool (`vision_wait ≈ ramplus`). CLIP (~190 ms) and faces (~17 ms) sit downstream and are *starved*, not throttled. The CUDA-pool fix was then TESTED on hardware (EP-aware sizing → pool=3) and **REGRESSED** throughput to 3.9 files/s (RAM++ 670→812 ms/file, RSS→7.6 GB) — 3 RAM++ Swin-L sessions over-subscribe the single GPU and thrash, proving RAM++ is **GPU-COMPUTE-bound, not concurrency-bound** (reverted). The only remaining win is a **batched RAM++ ONNX re-export** (~2-4×; needs an offline dynamic-axis export + a batch coordinator) or a lighter tagger. CUDA already binds, so the ceiling is RAM++ compute capacity, not the EP. (DirectML throughput was not exercised to completion this run — re-measure separately.)

## P1 — must fix

*P0 data-corruption blockers are listed first (they are the highest tier of "must fix"). De-duplicated refs cite every stream the issue appeared in.*

### P0 — cross-platform data corruption / shipping blockers

- **[PAR-1 / PAR-101 + ENG-3, ENG-4, ENG-5]** *v13 migration divergence — Windows-written DB fails to open on macOS.* `db/migrations.rs:40,68-71` ↔ `apple/.../Database.swift:332-339`. Windows registers/applies `v13_face_verification_anchors` (adds `face_a`/`face_b` to `face_verifications`); macOS's migrator stops at v12. GRDB validates `grdb_migrations` and throws on the unknown identifier, so a Win-upgraded library will not open on macOS. **Fix:** add the byte-identical v13 migration to macOS (`ALTER TABLE face_verifications ADD COLUMN face_a/face_b INTEGER`, exact identifier string); fix the stale Windows module-doc list (ENG-4) and `open_writer` "up to v7" comment (ENG-5). *Overlap: appears in engine (registry/doc) and parity (two refs) streams.*
- **[PAR-78]** *Scan tags written with different `source=` string — Library chips do not round-trip.* `dbwriter.rs:199` ↔ `apple/.../DBWriter.swift:322-323`. Windows writes scan tags `source='auto'`, macOS `source='vision'`; each platform's tile readers filter on its own value, so a cross-platform library shows **zero content-tag chips**. **Fix:** pin one canonical source token in the IPC schema; normalize existing rows.
- **[PAR-85]** *Timestamps use different epochs.* `scan_session.rs:139-154` ↔ `FileIDEngineMain.swift:425-427`. Windows writes UNIX-epoch f64; macOS writes Cocoa-reference-date (2001) f64 into the same DOUBLE columns. Values silently round-trip as the wrong wall-clock time (~31-year skew). **Fix:** pick one epoch (UNIX) on both sides.
- **[PAR-88 / PAR-121 + PAR-56]** *`vlm_model` model-kind strings differ.* `AppSettings.cs:112,133` / `deep_analyze.rs` ↔ `AIModels.swift:11-16`. Windows `qwen2_5_vl_7b`/`gemma_3_4b`; macOS `qwen2_vl_7b`/`gemma3_4b` (only `mistral_small_3_2` matches). A cross-platform open re-runs the entire VLM pass (skip-existing never matches) and the "analyzed" badge never lights. **Fix:** one canonical kind per VLM in the schema + a DB normalization migration. *Overlap: three parity refs (app-settings, deep-analyze, deepanalyze-view) — one root.*
- **[ENG-76]** *SHA256 verify-or-bail is inert — every registry entry has `sha256: None`.* `models/registry.rs:81-512`. Multi-GB GGUF/ONNX weights and native runtime DLLs are downloaded and executed/`LoadLibrary`'d with **zero integrity check**; the verify path is dead code despite the header claiming "SHA256-pinned manifest." **Fix:** pin real SHA256s in the registry/MODELS.md and make verification mandatory. *(Compounds ENG-78 part-resume corruption and ENG-82 sentinel fast-path.)*
- **[HW-1 — UNVERIFIED, reclassified out of P0]** *DirectML scan-to-completion was not measured.* The earlier "DirectML 0 files/s" observation came from a run that aborted at the model-presence gate due to a test-harness path bug (junction one level too high), NOT a product hang. The corrected run bound **CUDA** and completed cleanly. **Action:** re-measure a DirectML scan on the RTX 2060 (disable the CUDA pack) to confirm forward progress + throughput; only add a per-file CPU-fallback watchdog if a real stall is confirmed. STATE.md historically reports ~5–7 files/s on DirectML. *Needs hardware.*

### P1 — engine

- **[ENG-18]** *NTFS `file_ref` with high bit set aborts the entire scan.* `dbwriter.rs:240-289`. `Option<u64>` from the MFT reference exceeds `i64::MAX` once the sequence number is non-zero; rusqlite 0.31 `ToSql for u64` errors, the `?` propagates out of `flush`/`writer.run()`, and the **whole scan aborts** — every file in the batch and after is lost. Tests never trip it (tiny fixtures). **Fix:** bind `f.file_ref.map(|r| r as i64)` everywhere (lossless reinterpret); add a high-bit fixture. Confirm macOS stores inode identically.
- **[ENG-10 + ENG-38 + ENG-71]** *Unbounded image read/decode → OOM/DoS + allocation-abort crash.* `tagging.rs:749-784`, `deep_analyze.rs:251-266`, `vlm_server.rs:145-153`. Decoders `Vec::with_capacity(size_bytes)` + `read_to_end` with no size cap (×12 threads); a spoofed huge size aborts the process. The VLM server reads the whole file + base64 (1.33×) with no cap. **Fix:** clamp pre-reserve to a ceiling, use `Read::take(cap)`, add `image::io::Reader.limits()` pixel caps, reject oversized inputs as a per-file Err. *Overlap: three engine subsystems, one decode-bomb class.*
- **[ENG-42]** *Restructure no-op detected after uniquify, so a file already at its destination is renamed to " (2)".* `restructure_apply.rs:143-156`. `unique_destination` bumps the path before the `paths_equal` check, so re-planning an organized library churns every correct file (silently, in auto-file mode). **Fix:** check the no-op against the planned dest before uniquifying.
- **[ENG-49]** *In-flight DB-writing command tasks can re-dirty the WAL after the shutdown checkpoint and are never awaited.* `main.rs:359-385`. Fire-and-forget `tokio::spawn` handlers can COMMIT after `checkpoint_truncate` into a fresh, never-checkpointed WAL. *(Verifier errored — needs manual triage.)*
- **[ENG-50]** *Deep Analyze has no single-in-flight gate and shares one cancel flag across three commands.* `main.rs:606-648`. A second deep-analyze clears the shared cancel; a later cancel kills both batches; two batches race the VLM pool + writer. *(Verifier errored — manual triage.)*
- **[ENG-59]** *`ep_guard::reenable()` clears ANY disabled EP, not the one being re-enabled.* `ep_guard.rs:100-120`. Verifying/installing CUDA un-disables a crash-poisoned OpenVINO (and vice-versa); two EPs can never be disabled at once. *(Manual triage.)*
- **[ENG-60]** *QNN execution provider is never crash-guarded.* `ep_guard.rs:42-44`. A QnnHtp bind crash on Snapdragon crash-loops forever with no poison breadcrumb. *(Manual triage.)* *Needs hardware (Snapdragon).*
- **[ENG-66]** *BGE mean-pool indexes `attention_mask` by the model output seq length.* `bge_text.rs:118-122`. A re-export with a fixed/padded seq axis panics (index-OOB), crashing the BGE text path. *(Manual triage.)*
- **[ENG-69]** *`SFace.embed` does not assert 128-d output.* `sface.rs:92-102`. A wrong/quantized export yields off-dim embeddings that later panic in `cosine()` (release) or silently mis-cluster and diverge from the macOS DB. **Fix:** assert exact 128-d at embed; reject otherwise. *(Root-shared with PAR-2 below.)*
- **[ENG-72]** *Persistent-server VLM completion is not cancellable mid-call.* `vlm_server.rs:145-196`. Cancel only checked between files; a stalled completion runs up to the 300 s client timeout. *(Manual triage.)*
- **[ENG-77]** *Downloader performs no runtime host check.* `downloader.rs:165,372,548`. Only-HuggingFace egress is enforced solely by a build-time source grep; any non-literal URL can exfiltrate at runtime. *(Manual triage.)*
- **[ENG-78]** *Parallel resume blindly appends to stale `.part-NN` files with no length/integrity guard.* `downloader.rs:606-619`. Left-over parts from a different prior download are stitched into the final file; with SHA inert (ENG-76) corruption is silent and permanent. *(Manual triage.)*
- **[ENG-83]** *OCR runs WinRT on a tokio blocking thread that never initialized COM.* `shell/ocr.rs:55-74`. `OcrEngine` activation fails `CO_E_NOTINITIALIZED`; the error is swallowed to `Ok(None)`, so **OCR is silently dead on most/all scans**. **Fix:** `ensure_com_initialized()` like the video module. *Needs hardware (WinRT runtime).* *(Manual triage.)*
- **[ENG-84]** *HEIC decode invokes WinRT on the decoder thread which never initialized COM.* `shell/heic.rs:40-54`. Fails `CO_E_NOTINITIALIZED` even when the HEIF codec is installed, and is misreported to the user as "codec not installed." **Fix:** init COM; correct the error mapping. *Needs hardware.* *(Manual triage.)*
- **[ENG-88]** *Zip-bomb caps trust attacker-declared `entry.size()` instead of actual decompressed bytes.* `util/zip.rs:61-94`. `io::copy` streams with no real byte limit; a declared `size()=1` can expand to GB, filling the disk. *(Manual triage.)*
- **[ENG-97]** *`redact_path_for_log` passes through any path containing a `FileID` component.* `platform.rs:16-23`. A user file under any dir named `FileID` (e.g. a dev repo) is logged verbatim with the username — defeats the path-redaction invariant for a whole class of real paths. *(Manual triage.)*

### P1 — app

- **[APP-1]** *`UndoStack` mutates a non-thread-safe `LinkedList` from multiple threads with no lock.* `Services/UndoStack.cs:26-69`. `Push()` from the EngineClient "untrusted thread" races UI-thread `UndoAsync`/`Clear`/`TopLabel`; list-pointer corruption / `NRE` / `InvalidOperationException` escapes into the dispatcher with no `SafeRun` and can fast-fail the process. **Fix:** guard `_entries` with a private `lock` (or marshal `Push` to the captured UI dispatcher); keep the reverse `await` outside the lock.
- **[APP-2]** *No-progress watchdog captures a null UI dispatcher (`GetForCurrentThread()` after `ConfigureAwait(false)`).* `Services/ModelInstallerService.cs:600-611`. The 60 s stuck-install watchdog is silently disabled on the normal install path; a genuinely stuck download spins forever with no Retry. **Fix:** capture the UI dispatcher in the ctor (like `ModelSlot._ui`) and use that field.

### P1 — parity (db-incompat / win-regression / mac-gap with user impact)

- **[PAR-2 + PAR-31, PAR-32, PAR-33, PAR-34 + ENG-69, ENG-33]** *Face embedding is silently incompatible across platforms (SFace/arcface dim + alignment).* `dbwriter.rs:324-334` / `tagging.rs:1204` ↔ `FaceClustering.swift:402-431` / `:918`. Windows writes a **5-point similarity-aligned 128-d SFace** vector into `arcface_embedding`; macOS never aligns (axis-aligned padded bbox crop) and its clustering reads the column as **512-d ArcFace**. Face DBs do not cluster across platforms and the shared cosine thresholds misfire on macOS rows. macOS's `FaceAlign.align112` exists but has **zero callsites**, and macOS detection requests no 5-point landmarks at all. **Fix:** merge the macOS SFace lockstep — write the same 128-d LE bytes, wire `FaceAlign.align112` + `VNDetectFaceLandmarksRequest`, make `blobToEmbedding`/clustering dimension-agnostic; update the 512-d/2048-byte doc comments to 128-d/512-byte. *Overlap: 5 parity refs + 2 engine doc/assert refs — one face-embedding root.* *Needs a Mac.*
- **[PAR-3 + PAR-45, PAR-103, PAR-106 + PAR-102]** *macOS cannot persist "different people" verdicts (no `markPersonsDifferent` / no anchor key).* `bulk.rs:505-552,605-628` ↔ `PeopleView.swift:1296-1322` / `ClusterSuggestions.swift`. macOS has only a Merge button, never writes `face_verifications(same_person=0)`, and `findMergeSuggestions` is a `not_implemented_yet` stub — so rejected pairs re-suggest forever. **Fix:** add the negative-verdict UI + store write keyed on the stable `(min,max)` face-print anchor (the v13 mechanism), and exclude verified-different pairs. *Needs a Mac.*
- **[PAR-7]** *macOS `IPCCommand.Payload` is missing `markPersonsDifferent` and `wipeLibrary`.* `ipc/mod.rs:132-167` ↔ `IPCProtocol.swift:47-62`. macOS cannot encode either command. **Fix:** add both cases to the macOS enum. *Needs a Mac.*
- **[PAR-14 + PAR-15 + PAR-16]** *macOS never writes `content_hash`/`file_ref`, has no rename/move heal, and uses `INSERT OR REPLACE` that FK-cascades away all child rows on rescan.* `dbwriter.rs:284,507-589` ↔ `DBWriter.swift:287-317`. - **[PAR-26]** *macOS `FolderClassifier` (Anchor/Mixed/Junk, person/year/junk-denylist) has no faithful Windows port.* `restructure.rs:84-137` ↔ `FolderClassifier.swift:21-303`. - **[PAR-29]** *Photos year-month folder NAMES differ (Windows "March" vs macOS "03-Mar").* `restructure.rs:228-235` ↔ `Restructure.swift:302-306`. The same library lays out two incompatible trees; learn-your-style prototypes never match cross-platform. **Fix:** pick one canonical month format on both sides.

On 8 GB a user can launch Mistral (16 GB) and crash the engine. **Fix:** read physical RAM (Win32 `GlobalMemoryStatusEx`), port `fits()`, gate the cards + Analyze button + add the "needs more RAM" strip. *Headless-fixable on Windows.*
